# IDE에서 Coding Agent로 — 멘탈 모델과 작업 흐름

IntelliJ / Eclipse / VS Code 같은 IDE에 익숙한 사람이 Claude Code 같은 coding agent를 처음 쓸 때, 가장 먼저 바꿔야 하는 건 **작업 방식의 전제**입니다.

## 1. 멘탈 모델 전환

| 항목 | 기존 IDE | Coding agent |
|------|----------|--------------|
| 코드 작성 주체 | 내가 직접 친다 | 에이전트가 치고 내가 검토한다 |
| 작업 단위 | 키 입력·메서드 | 문장(prompt)·기능 단위 |
| 디버깅 | 디버거·브레이크포인트 | 로그·테스트·"왜 이렇게 했어?" 질문 |
| 코드 검색 | 인덱스·shortcut | 자연어 질문 (또는 파일 경로 직접 지시) |
| 리팩토링 | IDE refactor 메뉴 | "이 함수 추출해서 X로 옮겨줘" |
| 컨텍스트 | 모든 파일이 인덱싱돼 있음 | **에이전트는 읽은 파일만 안다** |
| 검토 시점 | 작성 직후 본인이 봄 | 작업 끝나고 **diff로 검토** |

### 가장 중요한 한 가지

**에이전트는 IDE처럼 모든 파일을 자동으로 인덱싱하지 않습니다.** 읽은 파일만 압니다. 그래서 작업 처음에 관련 파일·라인을 직접 알려주면 탐색 비용이 줄고 정확도가 올라갑니다.

## 2. 좋은 prompt 작성법

### 나쁜 예

> "로그인 기능 만들어줘"

너무 광범위해서 엉뚱한 방향으로 갑니다. 인증 방식, 토큰 저장 위치, UI 등 가정할 게 너무 많음.

### 좋은 예

> "`src/handlers/login.ts:42`의 `validateToken` 함수가 만료 토큰을 거부 안 함.
> JWT `exp` 클레임 검사 추가해줘. 테스트는 `__tests__/login.test.ts`에."

- **파일 경로·라인 명시**
- **현재 동작과 기대 동작 명시**
- **테스트 위치 등 제약 명시**

### Rule of thumb

> 동료 개발자에게 Slack DM 보낸다고 생각하고 컨텍스트를 충분히 주세요.
> "이 코드는 X 때문에 만든 거고, Y에서 호출되며, Z처럼 동작해야 함" — 이런 정보가 있어야 에이전트도 사람처럼 판단합니다.

## 3. 작업 크기에 맞는 흐름

### 작은 작업 — 곧장 지시

- 오타 수정, 한 줄 수정, 명확한 rename
- Normal mode 그대로, 그냥 지시 → 권한 승인

### 중간 작업 — 계획 받고 진행

- 한 파일 안에서 여러 함수 수정
- 단순 기능 추가
- 흐름:
  ```
  "이거 어떻게 고칠지 짧게 보여줘. 코드 변경은 아직 하지 마."
  → 계획 받음 → 승인 → 진행
  ```

### 큰 작업 — Plan mode

- 멀티 파일 리팩토링
- 새 기능 추가 (DB·API·UI까지)
- 잘 모르는 코드베이스에서의 변경
- 흐름은 [modes.md](modes.md)의 "패턴 A" 참고

## 4. TDD가 정말 잘 맞습니다

Coding agent가 가장 폭주하기 쉬운 순간은 **사람의 검증 포인트가 없는 긴 자율 실행**입니다.
TDD는 그 사이에 자연스러운 검증 포인트를 박아 줍니다:

```
1. "이 기능 구현 전, 어떤 케이스를 테스트해야 할지 test list 먼저 보여줘"
   → 사람이 검토. 빠진 케이스 추가, 불필요 제거
2. "OK, 이 테스트들을 실패하는 형태(red)로 먼저 작성해줘"
   → diff 검토. 테스트 자체가 의도대로 작성됐나?
3. "이제 테스트가 통과(green)하도록 구현해줘"
   → diff 검토. 구현이 합리적인가?
4. (선택) "리팩토링할 만한 부분 있어?"
```

각 단계에서 사람이 키를 잡고 있어 에이전트가 엉뚱한 방향으로 멀리 가지 못합니다.

## 5. 검토는 반드시 "diff 보기"로

- 에이전트가 만든 변경은 **항상 diff로 검토** 후 다음 작업 진행
- IDE의 git diff 뷰처럼 보이게 요청 가능: "지금까지 변경 diff 보여줘"
- 자동 승인 모드(Auto-accept)는 신뢰 쌓인 작업·반복 작업에서만
- 절대 통과시키지 말 것: 이해 안 되는 변경. **이해되지 않은 코드는 머지하지 않는다는 원칙은 사람 코드 리뷰와 동일**

## 6. 막혔을 때 쓰는 질문

- **"왜 이렇게 했어?"** — 가정의 오류 빨리 잡기
- **"다른 방법 있어? trade-off 비교해줘"** — 한 가지 답에 갇히지 않기
- **"내가 뭘 놓쳤을 가능성이 있어?"** — 사람이 모르는 가정 끌어내기
- **"이 코드 동작 단계별로 설명해줘"** — 디버깅 시작점

## 7. 에이전트가 잘하는 것 / 못하는 것

### 잘하는 것

- 정형화된 boilerplate 작성 (CRUD, 테스트 골격)
- 라이브러리·프레임워크 사용법 적용
- 코드 탐색·grep·"여기서 X 어디 쓰여?" 같은 조사
- 리팩토링 (특히 IDE refactor 메뉴로 안 되는 패턴)
- 문서화·주석·커밋 메시지 초안

### 못하는 것 (또는 사람 검증 필요한 것)

- 도메인 지식이 깊게 필요한 비즈니스 결정
- 외부 시스템과의 미묘한 인터페이스 (사람이 직접 확인 필요)
- 보안 critical 코드 — 사람의 명시적 검토 필수
- 본인이 모르는 사실을 그럴듯하게 만들어냄 (hallucination) → "확실해? 코드 읽어보고 답해" 가 효과적

## 8. 안전장치

### 자동으로 막히는 것 (Claude Code 기본)

- `git push --force`, `git reset --hard`, 브랜치 삭제 등 위험 작업 → 명시적 승인 없이는 실행 X
- 외부 시스템에 영향 주는 작업 → 사람 확인 우선

### 본인이 챙겨야 하는 것

- **커밋 시점**: 명시적으로 "커밋해줘"라고 할 때만 실행됨. 작업 도중 중간 저장 의미로 커밋하고 싶으면 직접 지시
- **브랜치 분리**: 큰 작업은 별도 브랜치에서 시작. 잘못돼도 main 보호
- **백업**: 중요한 변경은 작업 전 git stash 또는 임시 커밋

## 9. 자주 하는 실수

| 실수 | 결과 | 해결 |
|------|------|------|
| `/clear` 안 하고 다른 작업 시작 | 이전 작업 컨텍스트가 영향 → 엉뚱한 결정 | 새 작업마다 `/clear` |
| 파일 경로 안 알려줌 | 에이전트가 grep으로 헤맴 → 비효율 | `@<path>` 또는 경로 직접 명시 |
| Auto-accept 켜둔 채 큰 작업 시작 | 폭주 변경 발생 가능 | 큰 작업은 Plan mode로 시작 |
| 이해 안 되는 변경을 그냥 승인 | 나중에 디버깅 지옥 | 이해될 때까지 질문 |
| "다 해줘" 식 prompt | 광범위 변경 → 검토 불가능 | 단계별로 쪼개서 지시 |
| 테스트 안 돌리고 진행 | 회귀 누적 | 변경마다 관련 테스트 실행 |
| 메모리에 너무 많이 저장 | 컨텍스트 오염 | 정말 재사용될 정보만 저장 |

## 10. 일상 작업 사이클 예시

```
# 아침에 작업 시작
/clear
"오늘 할 일: foo 모듈 버그 수정. 먼저 src/foo.ts랑 관련 테스트 보고
어떤 식으로 접근하면 좋을지 알려줘."

# 계획 받음 → 검토 → 진행
"OK, 그 방향으로 가자. test list 먼저 보여줘."

# TDD 사이클
"red 테스트 작성"
"green되도록 구현"
"npm test 돌려서 결과 보여줘"

# 완료 후
/review        # 자체 리뷰
"diff 보여줘"
"커밋 메시지 초안 만들어줘"
"커밋해줘"

# 다른 작업으로 전환
/clear
...
```
